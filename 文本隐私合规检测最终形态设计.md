# 文本隐私合规检测最终形态设计

## 一、文档目的

本文档定义文本隐私合规检测的最终目标形态。

该目标形态服务于整个教育数据集建设流程，核心目标只有一个：

- 对清洗后的文本数据进行隐私合规治理，确保数据中不存在影响大模型训练的不合规隐私内容；
- 同时不破坏下游自动化标注流程的可执行性；
- 最终产出可用于教育等专有领域大模型训练的高质量合规数据。

本文档不再从“当前已经实现了什么”出发，而是从“最终应当如何设计”出发，统一说明：

- 文本隐私合规检测的标准流程；
- 每一步的输入、输出和判断目标；
- 如何做上下文分析；
- 如何做处理决策；
- 如何支持用户对自动化结果进行人工审核和修改。

## 二、总体设计原则

文本隐私合规检测不应只是“扫出隐私词”。

最终形态必须满足以下原则：

1. 先识别所有疑似隐私片段，再判断这些片段在当前语境下是否真正构成不合规风险。
2. 不能只输出结论，必须输出解释，讲清楚为什么合规或为什么不合规。
3. 不能只做片段级判断，还必须做文档级综合判断。
4. 不能只给“删或不删”两种结果，而应支持保留、脱敏、泛化、人工复核、排除训练等多种治理方式。
5. 自动化结果必须允许人工审核修改，且所有修改必须留痕可审计。
6. 前端展示应以用户可理解的自然语言解释为主，内部编码只作为辅助信息，不应成为主要交互语言。
7. 所有“上下文分析结果”都不能只以标签形式展示，而必须被组织成面向用户的口语化说明，明确讲清楚该片段出现在什么上下文中、为什么因此被判定为合规或不合规。
8. 隐私片段一旦被识别出来，默认应进入治理流程；上下文分析的首要作用，是把那些在当前场景下本可保留的片段从默认治理范围中减除，而不是因为缺少上下文就直接放行。

## 三、最终目标下的核心流程

文本隐私合规检测的核心主线应当是：

`隐私片段扫描 -> 片段级上下文分析 -> 片段级合规裁决 -> 文档级综合判断 -> 处理决策 -> 用户可视化复核与修改 -> 合规数据输出`

下面对每一步做详细定义。

## 四、默认治理与上下文减量原则

在最终形态中，系统必须先有一个稳定的默认治理基线，再允许上下文分析对这个基线做减量修正。

这套原则应明确如下：

1. 隐私信息一旦被扫描识别，默认进入治理流程，不因上下文缺失而直接放行。
2. 上下文分析的首要作用，是识别那些在当前场景下本可保留的片段，并将其从默认治理范围中减除。
3. 文档级上下文缺失时，不使用文档级信息进行豁免判断，系统退回片段级分析和默认治理策略。
4. 片段级上下文缺失或不足时，不应将该片段直接判定为合规可保留，而应按该类隐私信息的默认保守处理策略执行。
5. 上下文分析不仅影响“是否可保留”，也影响“应治理到何种程度”；对于已经确认的真实隐私信息，学生、未成年人、信息登记表、教育记录等语境可以提升处理强度。
6. 因此，上下文分析在“是否进入治理”上主要表现为减量机制，在“治理强度与流转限制”上仍可能发挥增强作用。

换言之，系统的判断逻辑不是“先靠上下文决定是不是要处理”，而应当是：

`先默认治理，再用上下文做保留豁免；若无足够上下文支持保留，则按默认保守策略处理。`

## 五、流程步骤与详细内容

### 4.1 数据接入与预处理

#### 目标

为隐私检测准备稳定、可定位、可审计的输入文本对象。

#### 输入

- 清洗后的 TXT、JSON、JSONL、CSV 抽取文本字段等
- 数据来源信息
- 数据集构建任务信息
- 场景信息，如教育数据、学生数据、未成年人场景等

#### 处理内容

1. 抽取实际检测文本。
2. 生成 `doc_id`、文本哈希、来源路径等基础标识。
3. 保留元信息：
   - 数据来源
   - 任务批次
   - 文档类型
   - 字段名
   - 上游清洗信息
4. 建立字符级偏移索引，保证后续所有隐私片段都能精确定位。

#### 输出

- 标准化文本对象
- 文档元信息
- 可供后续定位的字符偏移体系

### 4.2 文档级背景上下文构建

#### 目标

在扫描隐私片段前，先让系统知道“这是一份什么文本”。

#### 为什么需要这一步

同样一个片段，出现在不同文档里，风险完全不同。

例如：

- “张三”出现在小说示例中，风险较低；
- “张三，学号 STU-001，成绩 92”出现在学生档案中，风险显著提高。

#### 处理内容

系统应先构建文档级背景上下文，包括：

- 文档主题
- 文档类型
- 场景类型
- 主体类型
- 来源类型
- 是否属于教育数据或未成年人场景

#### 建议识别的上下文维度

1. 文档类型
   - 学生档案
   - 成绩记录
   - 家校沟通记录
   - 教材示例
   - 新闻报道
   - 制度公告
   - 测试样例
   - 模板文本

2. 数据主体
   - 学生
   - 未成年人
   - 家长
   - 教师
   - 普通成年人
   - 不明确主体

3. 数据来源
   - 用户上传
   - 内部文档
   - 公开网页
   - 公开教材
   - 系统样例
   - 测试数据

4. 使用目标
   - 进入训练数据集
   - 进入标注前自动处理链路
   - 暂存复核池

#### 输出

- 每个文档的背景上下文对象
- 后续片段分析所需的全局语境

#### 文档级上下文缺失时的默认处理原则

并不是所有文本都能稳定识别出可靠的文档级上下文。

因此，当系统无法判断一篇文本到底是学生档案、教材示例、公告通知还是测试样例时，不应因为“文档类型不明确”就自动把其中的隐私片段判定为可保留。

正确做法是：

- 文档级上下文存在时，用它辅助做保留豁免或增强治理强度；
- 文档级上下文缺失时，忽略这一层豁免能力，退回片段级分析与默认治理；
- 文档级上下文缺失本身，不应成为放行理由。

### 4.3 隐私信息片段扫描

#### 目标

由大模型扫描文本，尽可能完整地召回所有疑似隐私信息片段。

#### 为什么由大模型执行

隐私片段的表达形式高度多样，不只是固定格式匹配。

例如：

- “13800138000”是手机号；
- “王老师电话见家校联系册”可能间接指向联系方式；
- “成绩为 92 分”属于教育记录；
- “母亲微信已登记”可能涉及监护人联系信息。

大模型更适合做高召回扫描。

#### 扫描要求

大模型应对文本进行逐段扫描，识别所有疑似隐私片段，并输出结构化结果。

每个片段至少应包含：

- `fragment_id`
- `doc_id`
- `risk_type`
- `text`
- `start`
- `end`
- `confidence`
- `initial_explanation`
- `suggested_replacement`

#### 需要重点扫描的隐私类型

至少包括：

- 姓名
- 手机号 / 固定电话
- 邮箱
- 社交账号
- 身份证 / 护照 / 准考证 / 学号
- 地址 / 精确位置
- 成绩 / 排名 / 处分 / 学籍记录
- 家长 / 监护人联系方式
- 银行卡 / 支付账户
- 医疗 / 心理 / 健康记录
- 密钥 / Token / 凭据
- 未成年人相关敏感身份信息

#### 输出

- 疑似隐私片段列表

### 4.4 片段定位校验与标准化

#### 目标

保证扫描结果在原文中可精确定位，并统一风险类型表达。

#### 处理内容

1. 校验 `start/end` 是否与原文一致。
2. 对可自动纠偏的片段做偏移修正。
3. 对无法定位的结果标记为“待复核异常片段”。
4. 对风险类型做统一归一化，例如：
   - `phone` 归一为 `phone_number`
   - `token` 归一为 `secret`
   - `psychological_record` 归一为 `medical_record` 大类下的敏感记录

#### 输出

- 标准化隐私片段列表
- 无效定位片段列表

### 4.5 片段级上下文分析

#### 目标

对每一个隐私片段进行语义和场景判断，确定它是否真的构成不合规风险。

#### 这一层为什么是核心

隐私扫描只能回答：

- “这里像不像隐私信息”

上下文分析要回答的是：

- “这里是不是不合规隐私”
- “如果不合规，为什么不合规”
- “如果可保留，为什么可保留”
- “是否需要处理后才能进入数据集”

#### 上下文分析应如何进行

每一个隐私片段都应由大模型结合多层上下文完成分析。

分析时应至少使用以下信息：

1. 片段局部上下文
   - 片段前后若干字符
   - 所在完整句子
   - 所在段落
   - 是否有字段标签，如“姓名：”“电话：”“住址：”

2. 片段邻近语义
   - 周围是否同时出现学校、班级、成绩、家长、监护人等词
   - 是否与其他隐私片段组合出现

3. 文档级上下文
   - 文档整体是不是学生档案、成绩单、家校沟通、公告、教材示例、测试样例等

4. 来源上下文
   - 该文本是公开示例、内部记录、用户上传还是系统样例

5. 主体上下文
   - 数据主体是不是学生、未成年人、家长、教师

6. 训练影响判断
   - 该片段是否会直接影响训练数据合规性
   - 若保留或处理不充分，是否会把个人信息带入训练数据集

7. 自动标注影响判断
   - 该片段是否会影响下游自动标注流程
   - 是否会因为替换方式不当破坏标注结构

#### 片段级分析应输出什么

每个片段不应只输出一个“风险标签”，还应输出完整裁决结果：

- 是否属于真实隐私信息
- 是否构成当前场景下的不合规风险
- 是否可直接保留
- 是否需要脱敏或泛化
- 是否需要人工复核
- 裁决解释

#### 上下文分析结果不能只输出标签

这一点必须单独强调。

系统在做完上下文分析后，不能只给出类似下面这种机器化结果：

- 上下文类型：课本
- 主体类型：非真实人物
- 来源：公开网站
- 风险判断：合规

这种写法只适合内部结构化存储，不适合直接展示给用户，也不足以支撑用户理解和人工审核。

最终给用户看到的解释说明，必须是完整、自然、口语化、可直接理解的合规说明。也就是说，系统不仅要“识别出上下文标签”，还要把这些标签组织成一句或一段能让用户直接看懂的话。

解释说明至少应回答四个问题：

1. 这个隐私片段出现在什么内容里。
2. 这个上下文说明它更像是真实信息，还是示例信息、公开信息、模板信息。
3. 这个片段为什么会被判定为合规、需处理、或不合规。
4. 这个判断会对后续训练数据集和自动标注流程产生什么影响。

#### 解释说明的书写要求

每个片段的合规说明都应满足以下要求：

1. 必须引用具体上下文，而不是只写抽象标签。
2. 必须写成自然语言句子，而不是字段罗列。
3. 必须让用户看完后能直接理解“为什么这样判”。
4. 必须同时说明“上下文事实”与“合规结论”之间的关系。
5. 必须避免空洞表述，例如“命中隐私类型，因此不合规”这类没有上下文依据的话。

#### 片段级上下文不足时的默认处理原则

并不是每一个隐私片段都能找到充分的片段级上下文。

有些片段可能只是孤立出现在文本中，例如一个姓名、一串号码、一个地址短语，周围没有足够信息说明它到底是示例、真实对象还是公开信息。

在这种情况下，系统不应因为“缺少上下文”就把它判定为可保留，而应执行默认保守策略。

这里的默认保守策略不是所有片段统一做同一种处理，而是按隐私类型执行各自的默认治理动作。例如：

- 姓名、电话、邮箱：默认脱敏
- 地址：默认泛化或脱敏
- 学号、成绩记录：默认脱敏、受控流转或人工复核
- 身份证、银行卡、密钥、医疗记录：默认更严格处理，不应仅因上下文缺失而放行

因此，片段级上下文不足时的正确原则是：

- 不能据此判定其为合规可保留；
- 应按该类隐私信息的默认保守处理策略执行；
- 只有在上下文提供了充分的保留理由时，才可以把该片段从默认治理范围中减除。

#### 合规解释说明示例

以下示例代表最终系统应当输出的说明风格。

##### 示例一：姓名被判定为合规

不推荐写法：

- 上下文类型：课本
- 非真实姓名
- 结论：合规

推荐写法：

- “该姓名出现在课本讲解示例内容中，周围语句是在说明知识点，并不是学生档案、报名信息或家校记录中的真实身份信息，因此该姓名被判定为可保留的合规内容。”

##### 示例二：电话号码被判定为合规

不推荐写法：

- 来源：公开网站
- 电话类型：公开联系方式
- 结论：合规

推荐写法：

- “该电话号码出现在公开网站的机构联系方式栏目中，语义上属于对外公开的联系电话，而不是学生、家长或教师个人隐私联系方式，因此可判定为合规保留。”

##### 示例三：地址被判定为不合规

不推荐写法：

- 文档类型：信息登记表
- 主体：学生
- 风险：地址隐私
- 结论：不合规

推荐写法：

- “该地址出现在学生信息登记表中，并且与学生身份信息直接关联，属于可定位到具体个人的真实住址信息，会影响训练数据集的隐私合规性，因此不能原文保留，需进行泛化处理或排除出训练数据集。”

##### 示例四：成绩记录需人工复核

推荐写法：

- “该片段记录了具体成绩信息，文本整体看起来像学生学习记录，但当前内容无法完全确认它是公开示例还是真实学生记录。由于它可能属于学生教育记录，系统建议先进入人工复核，再决定是否脱敏后保留。”

#### 片段级裁决结果建议分为三类

1. 可直接保留
   - 片段虽像隐私，但在上下文中不构成训练风险
   - 例如公开示例名、测试样例值、非真实对象

2. 需治理后保留
   - 片段是真实隐私，不能原文进入数据集
   - 但经脱敏或泛化后仍可保留文本价值

3. 不可进入训练数据集
   - 片段风险高，无法仅靠局部处理解决
   - 或处理后仍会影响训练合规性

#### 每个片段必须给出解释说明

如果判定不合规，应解释：

- 该片段是什么类型的隐私信息
- 它为什么在当前上下文中构成风险
- 为什么不能原文进入训练数据集
- 如果需要处理，还应说明为什么需要脱敏、泛化、人工复核或排除训练

如果判定可保留，也应解释：

- 为什么虽然被扫描出来，但在当前上下文中仍然允许保留
- 是因为示例、模板、非真实主体，还是因为不具备识别性
- 并明确说明它为什么不会影响训练数据集和自动标注流程

### 4.6 文档级综合隐私判断

#### 目标

避免只看单个片段，忽视整篇文本组合后的风险。

#### 为什么必须做文档级判断

很多风险不是单片段就能看出来的，而是多个片段组合后才成立。

例如：

- 姓名
- 学号
- 成绩
- 家长电话
- 家庭住址

它们分开看都可以描述为“一个个片段风险”，但合在一起已经是一份高风险学生档案。

#### 文档级综合分析内容

1. 统计文档中出现的隐私类型组合。
2. 判断是否形成可重识别的个人画像。
3. 判断是否形成学生、未成年人、家长家庭信息的高风险聚合。
4. 判断即使做局部脱敏，整篇文本是否仍然不适合进入训练集。
5. 判断处理后的文本是否仍然适合进入自动化标注流程。

#### 文档级输出

文档级结果至少应包含：

- 最高风险级别
- 是否存在组合风险
- 是否适合进入训练数据集
- 是否适合进入自动标注流程
- 推荐治理动作
- 综合解释说明

### 4.7 处理决策生成

#### 目标

根据片段级和文档级判断，输出最终治理决策。

#### 重要说明

处理决策应拆成两层，不应混写。

第一层是内容处理动作，回答“文本怎么改”：

- `保留`
- `脱敏`
- `泛化`

第二层是流程处置动作，回答“数据怎么流转”：

- `正常流转`
- `人工复核`
- `排除训练`

如有必要，也可以保留中间状态：

- `受控流转`

但不建议把它作为面向用户的核心术语，前端更适合展示为：

- “需人工确认后进入后续流程”
- “暂不进入训练集”
- “脱敏后可进入自动标注链路”

#### 决策规则建议

#### 处理决策的基线原则

处理决策不是从“这段内容是否值得处理”开始，而是从“这段被识别出的隐私内容默认需要治理”开始。

随后再由上下文分析回答两个问题：

1. 这段内容是否有充分理由被判定为可保留？
2. 如果不能保留，应当按什么强度处理？

因此，决策生成必须遵循以下顺序：

1. 先按隐私类型给出默认保守处理动作。
2. 再根据片段级和文档级上下文判断是否可以减量放宽。
3. 如果上下文不足以支持放宽，则保持默认保守策略。
4. 如果上下文明确表明风险更高，则在默认治理基础上进一步提高处理强度或直接限制流转。

##### 情况一：可直接保留

适用条件：

- 片段被扫描到，但经上下文判断不构成真实训练风险
- 或文本本身属于示例、模板、测试值、非真实主体

处理：

- 内容处理：`保留`
- 流程处置：`正常流转`

##### 情况二：真实隐私，但可通过替换消除风险

适用条件：

- 属于姓名、电话、邮箱、家长联系方式、一般地址等
- 脱敏后仍保留文本语义价值
- 不影响下游自动标注结构

处理：

- 内容处理：`脱敏`
- 流程处置：`正常流转`

##### 情况三：精确信息不宜保留，但可降粒度保留语义

适用条件：

- 精确地址、具体学校门牌、精确位置等
- 不适合简单遮掉，但可以保留粗粒度语义

处理：

- 内容处理：`泛化`
- 流程处置：`正常流转`

##### 情况四：处理后仍需要人工判断

适用条件：

- 片段真实性不明确
- 示例与真实记录难区分
- 学生教育记录、未成年人家庭信息、组合风险较重
- 自动化处理可能误伤文本语义

处理：

- 内容处理：`保留`、`脱敏` 或 `泛化` 中任选其一作为机器建议
- 流程处置：`人工复核`

##### 情况五：不应进入训练数据集

适用条件：

- 证件号、银行卡、支付账号、密钥凭据
- 医疗心理记录
- 高风险未成年人识别信息
- 处理后仍明显可识别个人
- 整篇文档属于高风险学生隐私档案

处理：

- 内容处理：可给出参考性脱敏建议，但不能自动视为已放行
- 流程处置：`排除训练`

### 4.8 对下游自动标注的兼容性检查

#### 目标

确保隐私治理不会破坏后续自动化标注。

#### 为什么要单独做

文本隐私处理不只是合规问题，还会直接影响标注系统的稳定性。

例如：

- 替换词过长或结构异常，破坏分词和 span 对齐
- 删除整句导致标注目标丢失
- 替换方式不统一导致标注标签混乱

#### 检查内容

1. 替换占位符是否统一规范，如：
   - `<PERSON>`
   - `<PHONE>`
   - `<ADDRESS>`

2. 处理后字符结构是否仍适合标注。
3. 处理后是否破坏任务主语义。
4. 处理后是否还能支撑原本的标注任务。

#### 输出

- 是否允许进入自动标注链路
- 不允许时的原因说明

### 4.9 用户结果展示

#### 目标

让用户不仅看到“命中了什么”，还理解“为什么这样判”和“系统准备怎么处理”。

#### 每个隐私片段建议展示的内容

- 命中片段原文
- 风险类型
- 定位信息
- 上下文摘录
- 口语化上下文说明
- 合规判断结果
- 判断理由
- 建议处理动作
- 建议流程处置
- 替换预览
- 是否影响训练
- 是否影响自动标注

#### 文档级建议展示的内容

- 文档整体合规结论
- 是否可进入训练数据集
- 是否可进入自动标注流程
- 风险汇总
- 组合风险说明
- 文档级口语化合规说明
- 处理后预览

### 4.10 用户人工审核与修改

#### 目标

允许用户对所有自动化结果进行人工纠正，而不是只能被动接受系统输出。

#### 必须允许修改的内容

用户应能够对每一个隐私片段的自动化结果进行人工修改，包括：

- 是否真的是隐私片段
- 风险类型
- 片段定位范围
- 是否合规
- 不合规原因说明
- 合规原因说明
- 内容处理动作
- 流程处置动作
- 替换文本
- 是否允许进入训练数据集
- 是否允许进入自动标注链路

#### 用户还应能够进行的操作

- 新增遗漏的隐私片段
- 驳回误报片段
- 合并重复片段
- 拆分错误过大的片段
- 调整整篇文档的最终结论

#### 审核结果必须留痕

每次人工修改都必须记录：

- 修改前结果
- 修改后结果
- 修改人
- 修改时间
- 修改原因

这样才能保证：

- 审计可追踪
- 后续模型与规则可持续优化

### 4.11 最终输出产物

最终形态下，系统至少应输出以下结果：

1. 片段级扫描结果
2. 片段级上下文分析结果
3. 片段级合规裁决结果
4. 文档级综合判断结果
5. 内容处理结果
6. 流程处置结果
7. 脱敏或泛化后的文本预览
8. 用户人工审核结果
9. 完整审计日志

这些结果共同服务于：

- 平台前端展示
- 人工复核
- 下游自动标注
- 最终训练数据集导出
- 合规审计追踪

## 六、最终形态下的判断核心

文本隐私合规检测的最终核心，不是“有没有扫到隐私词”，而是以下三个问题：

1. 这段文本里有哪些疑似隐私信息片段？
2. 这些片段在当前上下文中，是否构成影响训练的数据合规风险？
3. 对这些风险做何种处理后，既能消除训练风险，又尽量不破坏下游自动标注和数据价值？

因此，最终形态下的文本隐私合规检测，必须是：

- 大模型负责高召回扫描；
- 大模型结合多层上下文做片段级分析；
- 系统再做文档级综合判断；
- 最终形成可解释、可处理、可人工修正的治理结果。

## 七、建议的最终结论体系

为了让前端和用户更易理解，最终对外展示不建议只用抽象代码，而应直接展示自然语言结论。

建议最终文档级结论使用以下三类：

1. `可直接进入数据集`
   - 无需处理，或已确认不构成隐私训练风险

2. `处理后可进入数据集`
   - 需要脱敏或泛化，处理后可继续进入自动标注与训练链路

3. `不可进入训练数据集`
   - 风险过高，或需要人工审核后另行决定

必要时再配套补充：

- 是否需人工复核
- 是否允许自动标注
- 是否已完成处理

## 八、结论

文本隐私合规检测的最终形态，不应是一个简单的“隐私命中器”，而应是一套完整的隐私治理系统。

它必须完成以下闭环：

`扫描 -> 理解 -> 判断 -> 解释 -> 处理 -> 展示 -> 人工修正 -> 审计留痕 -> 合规输出`

只有做到这一点，才能真正保证：

- 数据中不存在影响模型训练的不合规隐私内容；
- 隐私治理不会破坏下游自动化标注流程；
- 最终产出的数据能够稳定服务教育等专有领域大模型训练。
